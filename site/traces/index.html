🔗 Distributed Tracing: Seguindo o Caminho

Rastreie uma requisição através de todos os seus microsserviços

O que é Distributed Tracing?

Distributed tracing permite que você rastreie uma requisição única conforme ela percorre múltiplos serviços, bancos de dados e dependências. Você vê:

  • 📍 Quais serviços foram chamados em qual ordem
  • ⏱️ Quanto tempo cada um levou
  • ❌ Onde falhou (se falhou)
  • 🔗 Dependências entre serviços
  • 📊 Bottlenecks no fluxo

Exemplo: Uma Requisição em Microsserviços

Imagine que um usuário tenta comprar um item:

API Gateway

50ms
→

User Service

150ms
→

Inventory

120ms
→

Payment

300ms ⚠️
→

Order DB

80ms

Total: ~700ms | Bottleneck: Payment Service (300ms = 43%)

Conceitos Principais

Trace ID

Identificador único para uma requisição do início ao fim. Todos os serviços usam o mesmo trace_id.

trace_id: "abc123def456"

Span

Um "pedaço" da requisição — um serviço, operação ou query. Cada span tem início, fim e metadados.

span_id: "span_1", duration: 150ms

Parent-Child Relationship

Spans formam uma árvore. Se serviço A chama B que chama C:

A → B (parent) → C (child)

Instrumentação

Código que cria spans e propaga trace_id. Feito via bibliotecas (OpenTelemetry, Jaeger SDK).

tracer.startSpan("getUserData")

Baggage

Contexto que viaja com a requisição entre serviços. Ex: user_id, request_id.

baggage: { user_id: "123", tenant: "acme" }

Sampling

Não rastreie TUDO (muito caro). Amostre 10% ou trace erros sempre.

sampler: { type: "probabilistic", param: 0.1 }

OpenTelemetry: Padrão Open Source

OpenTelemetry (OTel) é o padrão emergente para observabilidade. Fornece:

  • ✅ SDKs para todas as linguagens
  • ✅ Padrão unificado (logs, métricas, traces)
  • ✅ Compatível com Jaeger, Zipkin, Datadog, etc
  • ✅ Sem vendor lock-in

Diferente de antes (cada ferramenta tinha seu SDK), agora você instrumenta uma vez com OTel e exporta para qualquer backend.

Ferramentas de Tracing

Jaeger

Open source, da Uber. Excelente UI, muito usado

docker run jaegertracing/all-in-one

Zipkin

Pioneer em distributed tracing. Simples, estável

docker run openzipkin/zipkin

OpenTelemetry

Padrão aberto. Funciona com qualquer backend

npm install @opentelemetry/api

Datadog APM

Profissional, pago. Melhor UX, correlação automática

gem install ddtrace

Exemplo: Instrumentação com OpenTelemetry

import { NodeTracerProvider } from '@opentelemetry/node';
import { JaegerExporter } from '@opentelemetry/exporter-jaeger';

const jaegerExporter = new JaegerExporter({
  endpoint: 'http://localhost:14268/api/traces',
});

const tracerProvider = new NodeTracerProvider();
tracerProvider.addSpanProcessor(
  new SimpleSpanProcessor(jaegerExporter)
);
tracerProvider.register();

// Uso em uma rota
app.get('/api/users/:id', async (req, res) => {
  const tracer = trace.getTracer('api-service');

  const span = tracer.startSpan('getUser');
  try {
    const user = await db.query('SELECT * FROM users WHERE id = ?', [req.params.id]);
    span.setAttributes({ user_id: user.id, status: 'success' });
    res.json(user);
  } catch (err) {
    span.recordException(err);
    span.setStatus({ code: SpanStatusCode.ERROR });
  } finally {
    span.end();
  }
});

Melhores Práticas

1️⃣ Correlacione Tudo

Use o mesmo trace_id em logs, métricas e traces:

{ trace_id: "abc123", span_id: "s1", userId: "123" }

2️⃣ Amostragem Inteligente

Rastreie 100% dos erros, 10% de sucessos (economize custo)

sampler: { errors: 1.0, successes: 0.1 }

3️⃣ Nomeie Spans Bem

Nomes descritivos e consistentes ajudam análise

db.query, http.request, cache.get (não: "operation")

4️⃣ Adicione Atributos Úteis

Contexto facilita debugging

{ http.status_code, db.statement, user_id }
