👨‍💻 SRE: Engenharia de Confiabilidade

Aplicando observabilidade para construir sistemas confiáveis

O que é um SRE?

Site Reliability Engineer (SRE) é um papel criado pelo Google. Não é um DevOps, não é um Ops tradicional — é um engenheiro de software que trabalha em operações.

"SRE is what happens when a software engineer is tasked with what is traditionally called operations."

— Google

Enquanto Ops manuais são reativos ("servidor caiu, conserta"), SRE é proativo: use engenharia para evitar problemas.

SRE vs DevOps vs Ops

Aspecto Traditional Ops DevOps SRE
Foco Keep systems running Deploy faster Reliable systems
Ferramenta preferida Runbooks manuais Ansible, Chef Código, automação
Resposta a falha Manual (1-2h) Scripted (minutos) Automatizada (segundos)
Foco em observabilidade Logs básicos Logs + alertas Logs + métricas + traces

Pilares de SRE

1️⃣ Observabilidade

Você só consegue consertar o que consegue ver. Logs, métricas, traces.

2️⃣ Automação

Tarefas repetitivas devem ser automatizadas. Deploy, scaling, recovery.

3️⃣ Disaster Recovery

Planejar para falhas. Backups, failover, caos engineering.

4️⃣ Capacity Planning

Entender crescimento e preparar infraestrutura adequadamente.

5️⃣ Change Management

Deploys controlados. Canary releases, blue-green, rollback.

6️⃣ Incident Response

Quando falha, responder rápido. Runbooks, escalação, postmortem.

SLI, SLO, SLA — A Tríade de Confiabilidade

SLI (Service Level Indicator)

O que você mede?

Uma métrica específica que mostra se seu serviço está saudável.

  • ✅ 99.9% de requests com sucesso
  • ✅ p99 latência < 200ms
  • ✅ Uptime > 99.95%

SLO (Service Level Objective)

O que você promete?

Meta que você se compromete a atingir (baseado em SLI).

  • ✅ 99.9% uptime por mês
  • ✅ 99% de requests rápidos
  • ✅ Máximo 0.1% de taxa de erro

SLA (Service Level Agreement)

O que você é penalizado?

Contrato legal. Se não atingir SLO, há multa/crédito.

  • ❌ Abaixo de 99.5% uptime? Cliente recebe 10% de desconto
  • ❌ Latência > 300ms? Multa por indisponibilidade

Error Budget: Aceitando Falhas

Error Budget é a quantidade de falha que você pode ter enquanto ainda atinge seu SLO.

Se seu SLO é 99.9% uptime por mês (30 dias):

  • Tempo total no mês: 30 × 24 × 60 = 43,200 minutos
  • Downtime permitido: 43,200 × 0.1% = 43.2 minutos
  • ✅ Você pode ter ~43 minutos de downtime/mês

Implicação: Se você já usou seu budget de 43 minutos neste mês, você é proibido de fazer deploy até o próximo mês. Risco demais.

Incident Response (Que fazer quando quebra?)

1️⃣ Detecção

Alertas disparam (baseado em métricas/SLI). Slack notifica.

2️⃣ Resposta Imediata

On-call SRE é acionado. Olha logs, métricas, traces.

3️⃣ Mitigação

Rollback de deploy? Reboot? Failover? Ação rápida.

4️⃣ Comunicação

Status page atualizado. Clientes informados.

5️⃣ Postmortem

Depois calmo, análise: "Por que falhou?" "Como evitamos próxima vez?"

6️⃣ Implementar Fix

Feature flag, circuit breaker, scaling automático, etc.
